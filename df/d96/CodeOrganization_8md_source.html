4 ## Overview
5 
6 The AliceO2 repository is subdivided in a number of sub-modules.
7 The _per detector_ sub-modules are grouped under the Detectors directory.
8 The _per function_ are in the top directory or grouped, e.g. Utilites.
9 
10 A typical submodule looks like :
11 ~~~~
12 .
13 |-- Common
14 | |-- CMakeLists.txt
15 | |-- doc
16 | | `-- All documentation files
17 | |-- include
18 | | `-- Common
19 | | `-- Factory.h
20 | |-- src
21 | | |-- Internal.h
22 | | `-- Factory.cxx
23 | `-- test
24 | `-- TestFactory.cxx
25 
26 ~~~~
27 
28 Depending on the case, some subdirectories can be voluntarily left out or added.
29 The headers go to the include directory if they are part of the interface, in the src otherwise.
30

32 
33 Other repositories in the AliceO2Group follow the same structure.

35 ## Principles
36 
37 A number of principles were agreed on that resulted in the above code organisation :
38 
39 * A _module_ is a set of code closely related sharing an interface that can result in one or more libraries.
40 * Favour is given to extracting large common components(modules/projects) into their own repositories within
41  AliceO2Group in github.
42 * AliceO2 therefore becomes a thinner repo containing :
43  * Detector specific code (e.g. related to reconstruction, simulation, calibration or qc).
44  * Commonalities (e.g. DataFormat, Steer-like), i.e. things other components depend on and that have not been extracted to their own repo.
45  * Global algorithms (e.g. global tracking), i.e. things that depend on several detectors.
46 * The directory structure can be either per detector or per function or a mixture.
47  The AliceO2 repository has a mixture of _per detector_ and _per function_ sub-modules with corresponding sub-structure.
48 * Dependencies are defined centrally as _buckets_.
49 * Each sub-module generates a single library linked against the dependencies defined in a single bucket.
50 * sub-modules' executable(s) link against the same bucket as the library and the library itself.
51 * Horizontal dependencies are in general forbidden (between sub-modules at the same level) (?)
52 * Naming : camel-case
53  * What is repeated / structural starts with a lower case letter (e.g. src, include, test).
54  * The rest (labels, unique names) start with an upper case letter (e.g. Common, Detectors).
55 * Why are headers in `MyModule/include/MyModule` and not directly in `MyModule/include ?`
56  * The difficulty here is that we have a number of constraints. First the headers must be installed in a directory
57  named after the module. Second the code which uses the headers must include `MyModule/xyz.h` and it must work
58  whether it is inside AliceO2 or in a different repo, i.e. whether the headers are installed or they are internal.
59  When evaluating the different options we ended up with this not-totally-perfect solution because all other solutions
60  broke one of the constraints or required a massive hurdle of CMake magic. If someone comes up with a different working
61  solution we would happily consider it.
