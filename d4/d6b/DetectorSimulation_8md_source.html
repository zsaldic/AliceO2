1 # Detector simulation documentation
2 
3 The present document collects information about the ALICE detector simulation executable and digitization procedure used in LHC Run3.
4 
5 ## Overview
6 
7 Detector simulation, the simulation of detector response from virtual particle events, consists of essentialy 2 parts:
8  a) the generation of simple (energy deposit) traces in the detector due to the passage of particles and the interaction with the detector material.
9  b) the conversion of those traces into (electronic) signals in the detector readout (usually called digitization).
10 
11 The first part is handled by the `o2sim` executable. The second part is handled in the `digitizer-workflow`.
12 
13 # Documentation of `o2sim`
14 
15 The purpose of the `o2sim` executable is to simulate the passage of particles emerging from a collision inside the detector and to obtain their effect in terms of energy deposits (called hits) which could be converted into detectable signals.
16 
17 ## Command overview
18 * **Basic help:** Help on command line options can be obtained with `o2sim --help`
19 * **Typical example:** A typical (exemplary) invocation is of the form
20 
21  ```o2sim -n 10 -g pythia8 -e TGeant4 -j 2 --skipModules ZDC,PHS```
22 
23  which would launch a simulation for 10 pythia8 events on the whole ALICE detector but ZDC and PHOS, using Geant4 on 2 worker processes.
24 * **Generated output**: The simulation creates at least the following files:

53 ## F.A.Q.
54 You may contribute to the documentation by asking a question
55 
56 #### 1. **How can I interface an event generator from ALIROOT**?
57 In order to access event generators from ALIROOT, such as `THijing` or `TPyhtia6`, you may use the `-g extgen` command line option followed by a ROOT macro setting up the event
58 generator. Examples thereof are available in the installation directory `$O2_ROOT/share/Generators/external`.
59 
60 For example, in order to simulate with 10 Hijing events, the following command can be run:
61 ```
62 o2sim -n 10 -g extgen --extGenFile $O2_ROOT/share/Generators/external/pythia6.C
63 ```
64 Macro arguments can be passed via
65 `--extGenFunc pythia6(14000., "pythia.settings")`.
66 
67 Users may write there own macros in order to customize to their needs.
68 
69 #### 2. **How can I run on a subset of geometry modules**?
70 Use the `--modules` or `-m` command line option. Example: `o2sim -m PIPE ITS TPC`
71 will run the simulation on a geometry/material consinsting of PIPE, ITS, TPC.
72 
73 #### 3. **How can I run with exactly the same events as used in an AliRoot simulation?**
74 
75 One may perform any arbitrary simulation with AliRoot and reuse the kinematics information in form of `Kinemtatics.root`
76 produced. The file contains primary and possibly secondary particles (added by transportation).
77 When the file is passed to `o2sim`, the primary particles my be used as the initial event.
78 Use the **`-g extkin`** command line option:
79 ```
80 o2sim -g extkin --extKinFile Kinematics.root ...
81 ```
82 
83 #### 4. **How can I generate events (signal) using the vertex position of already-generated (background) events?**
84 
85 This process might be called embedding, where one wants to merge two events generated independenly. For that to be physically correct, both events have to originate from the same interaction vertex.
86 Assuming that your already-generated (background) events are stored in the `o2sim.background.root` file, you can force the interaction vertex for the generation of a new set of events to be the same as the one in the background with the following command line option:
87 
88 ```
89 o2sim --embedIntoFile o2sim.background.root
90 ```
91 
92 Background events are sampled one-by-one until all events have been used. At that point the events start to be reused.
93 
94 #### 5. **How can I obtained detailed stepping information?**
95 Run the simulation (currently only supported in combination with `o2sim_serial`) with a preloaded library:
96 ```
97 MCSTEPLOG_TTREE=1 LD_PRELOAD=$O2_ROOT/lib/libMCStepLogger.so o2sim_serial -j 1 -n 10
98 ```
99 This will produce a file `MCStepLoggerOutput.root` containing detailed information about steps and processes (where, what, ...). The file can be analysed using a special analysis framework. See https://github.com/AliceO2Group/AliceO2/blob/dev/Utilities/MCStepLogger/README.md for more documentation.
