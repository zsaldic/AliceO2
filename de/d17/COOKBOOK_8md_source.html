31 #### How do I report failure for a given Algorithm?
32 
33 Whenever the driver process spots an error message, i.e. an error printed via `LOG(ERROR)` facility, when the driver process quits, it will exit with a exit code of 1. This includes any exception reported by the default exception handler.
34 
35 This comes handy, for example in tests.

37 ### Debugging on your laptop
38 
39 The way the DPL currently works is that the driver executable you launch,
40 will then take care of spawning one device per `DataProcessorSpec` in
41 a separate process. This means that in order to debug your code you need to
42 make sure gdb / lldb are actually debugging the right child process.
43 
44 For `gdb` you can use the `follow-fork-mode` setting. See
45 [here](https://sourceware.org/gdb/onlinedocs/gdb/Forks.html) for the full
46 documentation. This is unfortunately not available in
47 [lldb](https://bugs.llvm.org/show_bug.cgi?id=17972).
48 
49 Alternatively you can start your driver executable with the `-s` / `--stop`
50 command line option which will immediately stop execution of the children after
51 the fork, allowing you to attach to them, e.g. for gdb using:
52 
53 
54 ~~~~~~~{.sh}
55 
56 attach <pid>
57 ~~~~~~~
58 
59 or the `lldb` equivalent:
60 
61 
62 ~~~~~~~{.sh}
63 
64 attach -pid <pid>
65 ~~~~~~~
66 Note: On some systems, attaching might fail due to missing permission, and `gdb`
67 has to be started with `sudo`.
68 
69 In case you are building the DPL with the support for the debug GUI, you can
70 also attach a debugger to the running process by clicking on the
71 DataProcessorSpec you want to debug, which will show the Device inspector on
72 the right, and there click on the "Attach debugger" button. By default this
73 will start lldb in a Terminal.app window on mac, or GDB in an xterm elsewhere.
74 You can customise this behavior by setting the environment variable
75 `O2DPLDEBUG` to the command you want to run for the debugger and use the
76 environment variable `O2DEBUGGEDPID` to get the PID of the DataProcessor
77 currently selected. You can do this multiple times for all the devices you
78 wish to debug, but remember that you will need to quit the debugger if you want
79 DPL to exit.
80 
81 On linux you might need to start the debugger with `sudo` to have the permission
82 to attach, e.g. set O2DPLDEBUG to
83 
84 ~~~~~~~{.sh}
85 
86 export O2DPLDEBUG='xterm -hold -e sudo gdb attach $O2DEBUGGEDPID'
87 ~~~~~~~
88 Be sure to use single quotes to avoid direct expansion of O2DEBUGGEDPID variable.

104 ### Integrating with non-DPL devices
105 
106 Given the Data Processing Layer comes somewhat later in the design of O2, it's possible that you already have some topology of devices which you want to integrate, without having to port them to the DPL itself. Alternatively, your devices might not satisfy the requirements of the Data Processing Layer and therefore require a "raw" `FairMQDevice`, fully customised to your needs. This is fully supported and we provide means to ingest foreign, non-DPL FairMQDevices produced, messages into a DPL workflow. This is done via the help of a "proxy" data processor which connects to the foreign device, receives its inputs, optionally converts them to a format understood by the Data Processing Layer, and then pumps them to the right Data Processor Specs. In order to have such a device in your workflow, you can use the [`specifyExternalFairMQDeviceProxy`][specifyExternalFairMQDeviceProxy] helper to instanciate it. For an example of how to use it you can look at
107 [`Framework/TestWorkflows/src/test_RawDeviceInjector.cxx`][rawDeviceInjectorExample]. The `specifyExternalFairMQDeviceProxy` takes four arguments:
108 
109 
110 ~~~~~~~{.cpp}
111 
112 specifyExternalFairMQDeviceProxy("foreign-source",
113  {outspec},
114  "type=sub,method=connect,address=tcp://localhost:5450,rateLogging=1",
115  o2DataModelAdaptor(outspec, 0, 1)
116  ),
117 ~~~~~~~
118 
119 the first one is the usual `DataProcessorSpec` name, the second one is a list of outputs which we will create from the non-DPL device, the third one is a string to connect to the existing topology and a the fourth one is a function of the kind `o2::framework::InjectorFunction` which does the actual conversion. In this particular case we use the `o2DataModelAdaptor()` helper to create such an translation function since we know that our input is already respecting the O2 Data Model and most of the heavylifing can be done automatically.
120

127 ~~~~~~~

129 to create an out-of-band channel as specified in `channel-configuration` and hen use the `RawDeviceService` to get the raw FairMQDevice and send data hrough such a channel.

131 ## Customizing workflows creation (WIP)
132 
133 Sometimes it's handy to customise or generalise the workflow creation based on
134 external inputs. For example you might want to change the number of workers for
135 a given task or disable part of the topology if a given detector should not be
136 enabled.
137 
138 This can be done by implementing the function:
139 
140 
141 ~~~~~~~{.cpp}
142 
143 void customize(std::vector<o2::framework::ConfigParamSpec> &workflowOptions)
144 ~~~~~~~
145 
146 **before** including the `Framework/runDataProcessing.h` (this will most likely
147 change in the future). Each ConfigParamSpec will be added to the configuration
148 mechanism (e.g. the command line options) allowing you to modify them. Such options
149 will then be made available at workflow creation time via the `ConfigContext`
150 passed to the `defineDataProcessing` function, using the `ConfigContext::options()`
151 getter.

153 ## Completion policies (WIP)
154 
155 By default the data processing of a given record happens when all the fields of
156 that record are present. So if your Data Processor declares it will consume
157 `TRACKS` and `CLUSTERS`, for any given time interval both need to be produced
158 by some other data processor before the computation declared in yours can happen.
159 
160 Sometimes it's however desirable to customise such a behavior, so that some action
161 on the record can happen even if it's not complete. For example you might want
162 to start computing some quantity as a given message arrives and then complete the
163 computation once the record is complete. This is done by specifying by customising
164 the data processing CompletionPolicy. This can be done using the usual **Customization
165 mechanism** where a:
166 
167 
168 ~~~~~~~{.cpp}
169 
170 void customize(std::vector<CompletionPolicy> &policies);
171 ~~~~~~~
172 
173 function is provided before including `runDataProcessing.h`.
174 
175 Each `CompletionPolicy` requires the user to specify the `matcher` to select
176 which device is affected by it, and a `callback` to decide what action
177 expressed by a `CompletionOp` to take on a given input record.
178 
179 Possible actions include:
180 
181 * `CompletionPolicy::CompletionOp::Consume`: run the data processing callback and
182  mark the available fields in the input as consumed.
183 * `CompletionPolicy::CompletionOp::Process`: run the data processing callback, but do
184  not consume the field, which will be available when the next message for the field
185 * `CompletionPolicy::CompletionOp::Wait`: hold on the record but do not process it yet.
186 * `CompletionPolicy::CompletionOp::Drop`: drop the current available fields from the record.
187 
188 The default completion policy is to consume all inputs when they are all present.
189 
190 When the computation is dispatched with a partially completea `InputRecord`,
191 the user can check for the validity of any of its parts via the `InputRecord::isValid()`
192 API.

194 ## Customizing deployment configuration (WIP)
195 
196 By default every device instanciated by the Data Processing Layer connects to
197 the others using the PUB/SUB paradigm. This might or might not be desiderable
198 for some or even all of the connections. For this reason there is now a way to
199 customise the connections based on the ids of the devices being instanciated.
200 
201 In order to do so, one needs to implement the function
202 
203 
204 ~~~~~~~{.cpp}
205 
206 customize(std::vector<o2::framework::ChannelConfigurationPolicy> &policies)
207 ~~~~~~~
208 
209 **before** including `Framework/runDataProcessing.h` (this will most likely
210 change in the future). You can then extend the policies vector with your own
211 `ChannelConfigurationPolicy`. For each device to device edge, the system will
212 invoke the `ChannelConfigurationPolicy::match` callback with the ids of the
213 producer and of the consumer as arguments. If the callback returns `true`,
214 the `ChannelConfigurationPolicy::modifyInput` and
215 `ChannelConfigurationPolicy::modifyOutput` will be invoked passing the input and
216 output channel associated to the two devices, giving the opportunity to modify
217 the matching channels.

221 ## Lifetime support
222 
223 While initially foreseen in the design, Lifetime for Inputs / Outputs has not
224 yet being implemented correctly. However, once that happens, the following behaviors
225 will be implemented (naming foreseen to change). Once implemented it will be possible
226 to specify the following Lifetime types:
227 
228 * Timeframe: an input that gets processed only once.
229 * Condition: an input that is cache on considered valid for multiple computations,
230  according to its IOV.
231 * Transient: an output which is not actually sent. Useful to use the same mechanism
232  of the Message Passing API to create
233 * QA: an output which once send is also proposed as input to the subsequent computation,
234  allowing for accumulating data (e.g. histograms).
235 * SubTimeframe: an input which gets processed only once which has a
236  granularity of less than a timeframe. Within one computation
237  multiple of these can be created. They get sent as soon as
238  they go out of scope.

240 ## Wildcard support for InputSpec / OutputSpec
241 
242 In order to reduce the amount of code which one has to write to define inputs
243 and outputs, we plan to make the InputSpecs and OutputSpecs as veritable
244 matchers, supporting wildcards. For example if your Algorithm supports
245 processing clusters coming from multiple detectors, it will be possible to
246 specify:
247 
248 
249 ~~~~~~~{.cpp}
250 
251 InputSpec{"*", "CLUSTERS"}
252 ~~~~~~~
253 
254 If the user wants to get both clusters and tracks coming from the same detector,
255 it will be possible to write:
256 
257 
258 ~~~~~~~{.cpp}
259 
260 InputSpec{"*", "CLUSTERS"}, InputSpec{"*", "TRACKS"}
261 ~~~~~~~
262 
263 i.e. the first message which arrives will define the wildcard for all the other input
264 spec in the definition.

266 ### Data flow parallelism
267 
268 Data flow parallelism is simply expressed by tuning the data flow, adding explicitly the parallel data paths, using the appropriate `InputSpec` and `OutputSpec`.
269 
270 E.g.:
271 
272 
273 ~~~~~~~{.cpp}
274 
275 DataProcessorSpec{
276  "tpc_processor_1",
277  Inputs{},
278  Outputs{{"TPC", "CLUSTERS", SubSpec(0)}},
279  // ...
280 },
281 DataProcessorSpec{
282  "tpc_processor_2",
283  Inputs{},
284  Outputs{{"TPC", "CLUSTERS", SubSpec(1)}},
285  // ...
286 }
287 // ...
288 DataProcessorSpec{
289  "tpc_processor_18",
290  Inputs{},
291  Outputs{{"TPC", "CLUSTERS", SubSpec(17)}},
292  // ...
293 }
294 ~~~~~~~
295 
296 or alternatively the parallel workflows part could be generated programmatically:
297 
298 
299 ~~~~~~~{.cpp}
300 
301 parallel(
302  DataProcessorSpec{
303  "tpc_processor",
304  {InputSpec{"c", "TPC", "CLUSTERS"}}
305  },
306  18,
307  [](DataProcessorSpec &spec, size_t idx) {
308  spec.outputs[0].subSpec = idx; // Each of the 18 DataProcessorSpecs should have a different subSpec
309  }
310 )
311 ~~~~~~~
312 
313 Similarly this can be done for a component that merges inputs from multiple parallel devices, this time by modifying programmatically the `Inputs`:
314 
315 
316 ~~~~~~~{.cpp}
317 
318 // ...
319 DataProcessorSpec{
320  "merger",
321  mergeInputs({"a", "TST", "A", 0, Lifetime::Timeframe},
322  4,
323  [](InputSpec &input, size_t index) {
324  input.subSpec = index;
325  }
326  ),
327  {},
328  AlgorithmSpec{[](InitContext &setup) {
329  return [](ProcessingContext &ctx) {
330  // Create a single output.
331  LOG(DEBUG) << "Invoked" << std::endl;
332  };
333 }
334 // ...
335 ~~~~~~~
336 
337 When one declares a parallel set of devices you can retrieve the rank (i.e. parallel id) or the number of parallel devices by using the `ParalleContext`, which can be retrieved from the `ServiceRegistry` (see also the `Services` section below), e.g.:
338 
339 
340 ~~~~~~~{.cpp}
341 
342 size_t whoAmI = services.get<ParallelContext>().index1D();
343 size_t howManyAreWe = services.get<ParallelContext>().index1DSize();
344 ~~~~~~~

346 ### Time pipelining
347 
348 A second type of parallelism is time based pipelining. This assumes that the data can be subdivided in subsequent "time periods" that are independent one from the other and which are each identified by some timestamp entity. In this particular case it could result handy that some part of the workflow are actually processing different time periods. This can be expressed via the `timePipeline`, directive, e.g.:
349 
350 
351 ~~~~~~~{.cpp}
352 
353 // ...
354 timePipeline(DataProcessorSpec{
355  "processor",
356  {InputSpec{"a", "TST", "A"}},
357  {OutputSpec{"TST", "B"}},
358  AlgorithmSpec{[](ProcessingContext &ctx) {
359  };
360  }
361 }, 2);
362 // ...
363 ~~~~~~~
364 
365 which will result in two devices, one for even time periods, the other one for
366 odd timeperiods.
