1 # Data Processing Layer in O2 Framework
2 
3 ## Status quo and motivation for an O2 Data Processing Layer
4 
5 FairMQ currently provides a well documented and flexible framework for an actor based computation where each of the actors listens for message-like entities on channels and executes some code as a reaction. The key component which controls this is called a `FairMQDevice` (or *device* from now on) which can use different kind of transports to receive and send messages. In the most generic case, users are allowed to have full control on the state machine governing the message passing and have complete control on how the message handling is done. This of course covers all ALICE usecases in a generic manner, at the cost of extra complexity left to the user to implement. In most cases however a simplified way of creating devices is provided, and the user will simply create its own `FairMQDevice`-derived class, which registers via the `OnData(FairMQParts &parts)` method a callback that is invoked whenever a new message arrives. This however still holds the user responsible for:
6 
7 * Verifying that the required inputs for the computation are all available, both from the actual data flow (being it for readout, reconstruction or analysis) and from the asynchronous stream (e.g. alignment and calibrations).
8 * Create the appropriate message which holds the results and send it.
9 * Ensure the ease of testability, the code reuse and the proper documentation `OnData` callback. In particular there is no way to inspect which data is expected by a device and which data is produced.
10 
11 This is by design, because the FairMQ transport layer should not know anything about the actual data being transferred, while all the points above require some sort of inner knowledge about the data model and the data being moved around.
12 
13 The aim is to achieve the following:
14 
15 * **Explicit data flow:** Input and outputs are declared upfront and can be used for documentation or for automatic topology creation (assuming the actual processing environment is known).
16 * **Transport agnostic data processing:** users will not have to know about the details of how the data materialises on their device, they will always be handed the full set of payloads they requested, even if they come at different time.
17 * **Composability of data processing:** different process functions can in principle be chained and scheduled together depending on the desired granularity for devices.
18 
19 ## Separating data-processing from transport
20 
21 For the reasons mentioned above, we propose that the one of the developments which should happen with the O2 Framework work package is the development of a “Data Processing layer” which actually knows about the O2 Data Model (and for this reason cannot be part of FairMQ itself) and exploits it to validate, optimise and correctly schedule a computation on a user specified set of inputs.
22 
23 The Data Processing Layer in particular requires:
24 
25 * That the inputs types of each computation are provided upfront.
26 * That the outputs types of each computation are provided upfront.
27 * That a time identifier can be associated to inputs
28 
29 and given these premises it actually guarantees:
30 
31 * That whenever a new input arrives a `CompletionPolicy` is executed to decide wether the associate record is complete. By default such a `CompletionPolicy` waits for all the specified parts to have arrived.
32 * That no message passing happens during the performing of the computation, but uniquely at the end.
33 
34 ### Instanciating a workflow
35 
36 The description of the computation in such the Data Processing Layer is done via instances of the `DataProcessorSpec` class, grouped in a so called `WorkflowSpec` instance. In order to provide a description a computation to be run, the user must implement a callback which return an filled `WorkflowSpec`. E.g.:
37 
38 
39 ~~~~~~~{.cpp}
40 
41 #include "Framework/Utils/runDataProcessing.h"
42 
43 WorkflowSpec defineDataProcessing(ConfigContext &context) {
44  return WorkflowSpec {
45  DataProcessorSpec{
46  ...
47  },
48  DataProcessorSpec{
49  ...
50  }
51  };
52 };
53 ~~~~~~~
54 
55 See next section, for a more detailed description of the `DataProcessorSpec` class. The code above has to be linked into a single executable together with the Data Processing Layer code to form a so called driver executable which if run will:
56 
57 * Map all `DataProcessorSpec` to a set of `FairMQDevice`s (using 1-1 correspondence, in the current implementation).
58 * Instanciate and start all the devices resulted from the previous step.
59 * (Optionally) start a GUI which allows to monitor the running of the system.
60 
61 The `ConfigContext` object being passed to the function contains a set of user provided options to customise the creation of the workflow. For example you might want to change the number of workers for a given task or disable part of the topology if a given detector should not be enabled.
62 
63 In order to specify which options are to be present in the `ConfigContext`, the user can define the extension point:
64 
65 
66 ~~~~~~~{.cpp}
67 
68 void customize(std::vector<o2::framework::ConfigParamSpec> &workflowOptions)
69 ~~~~~~~

71 ### Describing a computation
72 
73 The description of the computation in such a layer is done via a `DataProcessorSpec` class, which describes some sort of processing of a (set of) O2 Data Payloads (*payloads* from now on), as defined by the O2 Data Model, eventually producing new payloads as outputs. The inputs to the computation, the outputs and the actual code to run on the former to produce the latter, is specified in a `DataProcessorSpec` instance. Multiple `DataProcessorSpec` instances can be grouped together in a `WorkflowSpec`to the driver code which maps them to processing devices accordingly.
74 
75 The `DataProcessorSpec` is defined as follows:
76 
77 
78 ~~~~~~~{.cpp}
79 
80 struct DataProcessorSpec {
81  using InitCallback = std::function<ProcessCallback(InitContext &)>;
82  using ProcessCallback = std::function<void(ProcessingContext &)>;
83  using ErrorCallback = std::function<void(ErrorContext &)>;
84  std::vector<InputSpec> inputs;
85  std::vector<OutputSpec> outputs;
86  std::vector<ConfigParamSpec> configParams;
87  std::vector<std::string> requiredServices;
88  AlgorithmSpec algorithm;
89 };
90 ~~~~~~~
91 
92 the inputs field represents a set of subscriptions to some kind of input data we would like to process. As per O2 Data Model
93 there is 3 major properties O2 attaches to a message: its origin, a description and a generic subspecification.
94 
95 So:
96 
97  {InputSpec{"clusters", "TPC", "CLUSTERS", 0}}
98 
99 really means "subscribe in input to the messages which have origin "TPC", contain objects of kind "CLUSTERS" and have a generic
100 subspecification 0 (e.g. to index the sector). In your code you will be able to retrieve these kind of messages using the "clusters"
101 label (see later the description of the InputRecord API).An InputSpec is effectively a three dimensional selection on space defined by all the possible origins, descriptions and subspecifications.
102 
103 Notice that the method `o2::framework::select` can be used to provide the query
104 in a more compact (and flexible way). Using such an helper the above becomes:
105 
106  select("clusters:TPC/CLUSTERS/0")
107 
108 Similarly the `OutputSpec` is a description of what the DataProcessor will produce, again in terms of (origin, description, subspecification) properties.
109 
110 The `configParams` vector would be used to specify which configuration options the data processing being described requires:
111 
112 
113 ~~~~~~~{.cpp}
114 
115 struct ConfigParamSpec {
116  std::string name;
117  enum ParamType type;
118  variant defaultValue;
119  ...
120 };
121 ~~~~~~~
122 
123 command line / configuration options would be automatically generated by it. These are available only at init stage, and can be used to configure services. They are not available to the actual `process` callback as all the critical parameters for data processing should be part of the data stream itself, eventually coming from CCDB / ParameterManager.
124 
125 Similarly the `requiredServices` vector would define which services are required for the data processing. For example this could be used to declare the need for some data cache, a GPU context, a thread pool.
126 
127 The `algorithm` property, of `AlgorithmSpec` is instead used to specify the actual computation. Notice that the same `DataProcessorSpec` can use different `AlgorithmSpec`. The rationale for this is that while inputs and outputs might be the same, you might want to compare different versions of your algorithm. The `AlgorithmSpec` resembles the following:
128 
129 
130 ~~~~~~~{.cpp}
131 
132 struct AlgorithmSpec {
133  using ProcessCallback = std::function<void(ProcessingContext &)>;
134  using InitCallback = std::function<ProcessCallback(InitContext &)>;
135  using ErrorCallback = std::function<void(ErrorContext &)>;
136 
137  InitCallback onInit = nullptr;
138  ProcessCallback onProcess = nullptr;
139  ErrorCallback onError = nullptr;
140  ...
141 };
142 ~~~~~~~
143 
144 The `onProcess` function is to be used for stateless computations. It’s a free function and it’s up to the framework to make sure that all the required components are declared upfront. It takes as input the context for the current computation in the form of a `ProcessingContext &` instance. Such a context consist of:
145 
146 * An `InputRecord` which allows retrieving the current inputs matching the provided specification.
147 * A `ServiceRegistry` referencing the set of services it declared as required the computation.
148 * A `DataAllocator` allocator which can allocate new payloads only for the types which have been declared as `outputs`.
149 
150 #### Stateful processing
151 
152 `onProcess` is useful whenever your computation is fully contained in your input. In several cases, however, a computation requires some ancillary state, which needs to be initialised only on (re-)start of the job. For example you might want to initialise the geometry of your detector or open a file for reading. To do so, you can use the `onInit` callback, create the state in it and pass it to the returned `ProcessCallback` as captured arguments.
153 
154 E.g:
155 
156 
157 ~~~~~~~{.cpp}
158 
159 AlgorithmSpec{
160  InitCallBack{[](InitContext &setup){
161  auto statefulGeo = std::make_shared<TGeo>();
162  return [geo = statefulGeo](ProcessingContext &) {
163  // do something with geo
164  };
165  }
166  }
167 }
168 ~~~~~~~

170 ### Task based API
171 
172 The previous API is flexible enough to work for large variety of cases, including
173 creating `onProcess` callback on the fly depending on the `onInit` parameters.
174 .
175 
176 However, very often what the user wants to do is to initialise some state and
177 and invoke a "run" method. For this we provide a `Task` based API. In order to do so,
178 you need to inherit your task from `o2::framework::Task` and use the adaptor:
179 
180 
181 ~~~~~~~{.cpp}
182 
183 adaptFromTask<TASK>(task constructor arguments) -> AlgorithmSpec
184 ~~~~~~~
185 
186 to create the `AlgorithmSpec`. A full example can be found in
187 `Framework/Core/test/test_Task.cpp`

193 ### Using inputs - the `InputRecord` API
194 
195 Inputs to your computation will be provided to you via the `InputRecord`. An instance of such a class is hanging from the `ProcessingContext` your computation lambda is passed and contains one value for each of the `InputSpec` you specified. E.g.:
196 
197 
198 ~~~~~~~{.cpp}
199 
200 InputRecord &inputs = ctx.inputs();
201 ~~~~~~~
202 
203 From the `InputRecord` instance you can get the arguments either via their positional index:
204 
205 
206 ~~~~~~~{.cpp}
207 
208 DataRef ref = inputs.getByPos(0);
209 ~~~~~~~
210 
211 or using the mnemonics-label which was used as first argument in the associated `InputSpec`.
212 
213 
214 ~~~~~~~{.cpp}
215 
216 DataRef ref = inputs.get("points");
217 ~~~~~~~
218 
219 You can then use the `DataRef` `header` and `payload` raw pointers to access the data in the messages.
220 
221 If the message is of a known messageable type, you can automatically get the content of the message by passing type T as template argument. The actual operation depends on the properties of the type. Not all types are supported, in order to get an object with pointer-like behavior, T has to be a pointer (T = U*).
222 
223 
224 ~~~~~~~{.cpp}
225 
226 auto p = args.get<T>("input");
227 ~~~~~~~
228 The return type is
229 
230 * `T const&` if `T` is a messageable type
231 * `T` if `T` is a `std::container` of a ROOT-serializable type
232 * `smart_pointer<T>` if `T` is a ROOT-serializable type and `T*` is passed
233 
234 Examples:
235 
236 * for messageable types there is no additional copy involved, the content is only for reading.
237 
238 
239 ~~~~~~~{.cpp}
240 
241 XYZ const& p = args.get<XYZ>("points");
242 ~~~~~~~
243 
244 * ROOT-serialized objects are automatically deserialized and returned as a smart pointer. Note that the requested type has to be pointer.
245 
246 
247 ~~~~~~~{.cpp}
248 
249 auto h = args.get<TH1*>("histo");
250 h->Print();
251 ~~~~~~~
252 
253 * container of ROOT-serialized objects are automatically deserialized and returned as container object.
254 
255 
256 ~~~~~~~{.cpp}
257 
258 auto points = args.get<std::vector<TPoint>>("points");
259 for (auto& point : points) {}
260 ~~~~~~~
261 
262 Check next section for known types. The framework will also take care of necessary deserialization.

264 ### Creating outputs - the DataAllocator API
265 
266 In order to prevent algorithms to create data they are not supposed to create, a special `DataAllocator` object is passed to the process callback, so that only messages for declared outputs can be created. A `DataAllocator` can create Framework owned resources via the `make<T>` method. In case you ask the framework to create a collection of objects, the result will be a `gsl::span` wrapper around the collection. A `DataAllocator` can adopt externally created resources via the `adopt` method. A `DataAllocator` can create a copy of an externally owned resource via the `snapshot` method.
267 
268 Currently supported data types for `make<T>` are:
269 
270 * Vanilla `char *` buffers with associated size: this is the actual contents of the FairMQ message.
271 * Messageable types: trivially copyable, non-polymorphic types.
272  These get directly mapped on the message exchanged by FairMQ and are therefore "zerocopy" for what the Data Processing Layer is concerned.
273 * Collections of messageable types, exposed to the user as `gsl::span`.
274 * TObject derived classes.
275  These are actually serialised via a TMessage and therefore are only suitable for the cases in which the cost of such a serialization is not an issue.
276 
277 Currently supported data types for `snapshot` functionality, state at time of
278 calling `snapshot` is captured in a copy, and sent when processing is done:
279 
280 * Messageable types.
281 * ROOT-serializable classes, serialised via a `TMessage`.
282  Classes implementing ROOT's `TClass` interface and std containers of those are automatically detected. ROOT-serialization can be forced using type converter `ROOTSerialized`, e.g. for types which can not be detected automatically
283 * `std::vector` of messageable type, at receiver side the collection is exposed as `gsl::span`.
284 * `std::vector` of pointers to messageable type, the objects are linearized in th message and exposed as gsl::span on the receiver side.

286 The `DataChunk` class resembles a `iovec`:
287 
288 
289 ~~~~~~~{.cpp}
290 
291 struct DataChunk {
292  char *data;
293  size_t size;
294 };
295 ~~~~~~~
296 
297 however, no API is provided to explicitly send it. All the created DataChunks are sent (potentially using scatter / gather) when the `process` function returns. This is to avoid the “modified after send” issues where a message which was sent is still owned and modifiable by the creator.

299 ### Error handling
300 
301 When an error happens during processing of some data, the writer of the `process` function should simply throw an exception. By default the exception is caught by the `DataProcessingDevice` and a message is printed (if `std::exeception` derived `what()` method is used, otherwise a generic message is given). Users can provide themselves an error handler by specifying via the `onError` callback specified in `DataProcessorSpec`. This will allow in the future to reinject data into the flow in case of an error.
302 
303 ### Services
304 
305 Services are utility classes which `DataProcessor`s can access to request out-of-bound, deployment dependent, functionalities. For example a service could be used to post metrics to the monitoring system or to get a GPU context. The former would be dependent on whether you are running on your laptop (where monitoring could simply mean print out metrics on the command line) or in a large cluster (where monitoring probably means to send metrics to an aggregator device which then pushes them to the backend.
306 
307 Services are initialised by the driver code (i.e. the code included via `runDataProcessing.h`) and passed to the user code via a `ServiceRegistry`. You can retrieve the service by the type of its interface class. E.g. for Monitoring you can do:
308 
309 
310 ~~~~~~~{.cpp}
311 
312 #include <Monitoring/Monitoring.h>
313 // ...
314 auto service = ctx.services().get<Monitoring>(); // In the DataProcessor lambda...
315 service.send({ 1, "my/metric" }); ...
316 ~~~~~~~
317 
318 Currently available services are described below.
319 
320 #### ControlService
321 
322 The control service allow DataProcessors to modify their state or the one of their peers in the topology. For example if you want to quit the whole data processing topology, you can use:
323 
324 
325 ~~~~~~~{.cpp}
326 
327 #include "Framework/ControlService.h"
328 //...
329 auto ctx.services().get<ControlService>().readyToQuit(true) // In the DataProcessor lambda
330 ~~~~~~~
331 
332 #### RawDeviceService
333 
334 This service allows you to get an hold of the `FairMQDevice` running the DataProcessor computation from with the computation itself. While in general this should not be used, it is handy in case you want to integrate with a pre-existing `FairMQDevice` which potentially does not even follow the O2 Data Model.
335 
336 #### Monitoring service
337 
338 Integration with the monitoring subsystem of O2 happens by getting the `o2::monitoring::Monitoring` interface. A simple example is:
339 
340 
341 ~~~~~~~{.cpp}
342 
343 #include <Monitoring/Monitoring.h>
344 // ...
345 auto service = ctx.services().get<Monitoring>(); // In the DataProcessor lambda...
346 service.send({ 1, "my/metric" }); ...
347 ~~~~~~~

349 for the full API documentation please have a look at:
350 
351 https://github.com/AliceO2Group/Monitoring
352 
353 Some suffix for the metrics are reserved to represent vector and tabular metrics.
354 
355 * `<some-metric-name>/n` contains the size of a vector metric at a given moment.
356 * `<some-metric-name>/m` contains the secondary size of a matrix metric at a given moment.
357 * `<some-metric-name>/<i>` where `<i>` is an integer contains the values of the i-th element in a vector metric or of the `<i>%n` column, `<i>/m` row of a matrix metric.
358 
359 #### InfoLogger service
360 
361 Integration with the InfoLogger subsystem of O2 happens in two way:
362 
363 * Explicitly, by getting the `AliceO2::InfoLogger::InfoLogger` service from the registry, like done in the monitoring case:
364 
365 
366 ~~~~~~~{.c}
367 ++
368 #include <InfoLogger/InfoLogger.hxx>
369 //...
370 auto logger = context.services().get<InfoLogger>(); // In the DataProcessor lambda
371 ~~~~~~~

373 * Implicitly, by using the standard FairLogger `LOG` macro. In order to enable this
374  the user needs to specify the minimum severity to send to the `InfoLogger` via the
375  `--infologger-severity` option, e.g.:
376 
377 
378 ~~~~~~~{.sh}
379 
380 ... --infologger-severity WARNING
381 ~~~~~~~
382 
383 Finally, one can configure the bahavior of the InfoLogger by using the `--infologger-mode` option.

385 #### Callback service
386 
387 A service that data processors can register callback functions invoked by the framework at defined steps in the process flow. This allows you to have customisation points for the following event:
388 
389 * `CallbackService::Id::Start`: before entering the running state.
390 * `CallbackService::Id::Stop`: before exiting the running state.
391 * `CallbackService::Id::Reset`: before resetting the device.
392 
393 Moreover in case you want to process events which are not coming from `FairMQ`, there is a `CallbackService::Id::ClockTick` which is called according to the rate specified for the backing FairMQ device.
394 
395 Similarly the `CallbackService::Id::Idle` callback is fired whenever there
396 was nothing to process.

398 ## Expressing parallelism
399 
400 If we want to retain a message passing semantic and really treat shared memory as yet another transport, we need to be very careful in how to express parallelism on data, so that the “single ownership model” of message passing forces us to either duplicate streams that need to be accessed in parallel, or to serialise workers which need to access the same data. Solutions like reference counting shared memory would not be allowed in such a scenario and in any case would require extra caution and support to make sure that failures do not leave dangling reference around (e.g. when one of the parallel workers abruptly terminates). First of all let’s consider the fact that there are two level of parallelisms which can be achieved:
401 
402 * Data flow parallelism: when data can be split in partitions according to some subdivision criteria (e.g. have one stream per TPC sector and have one worker for each).
403 * Time flow parallelism: when parallelism can be achieved by having different workers handle different time intervals for the incoming data. (e.g. worker 0 processes even timeframes, worker 1 processes odd timeframes).
404 
405 In order to express those DPL provides the `o2::framework::parallel` and `o2::framework::timePipeline` helpers to avoid expressing those explicitly in the workflow.

407 ## Integrating with pre-existing devices
408 
409 It can actually happen that you need to interface with native FairMQ devices, either for convenience or because they require a custom behavior which does not map well on top of the Data Processing Layer.
410 
411 This is fully supported and the DPL provides means to ingest foreign, non-DPL `FairMQDevice` produced messages into a DPL workflow. This is done via the help of a "proxy" data processor which connects to the foreign device, receives its inputs, optionally converts them to a format understood by the Data Processing Layer, and then pumps them to the right Data Processor Specs.
412 
413 This is done using the `FairMQRawDeviceService` which exposes the actual device on which an Algorithm is running, giving the user full control.

416 ## Customisation of default behavior
417 
418 The DPL aims at solving with the default behavior the majority of common usecases. For all the special cases to be handle, we however provide multiple customisation entrypoints, in particular to:
419 
420 * Change the policy for when an input record can be processed and what to do with the record afterwards. This is referred as `CompletionPolicy`.
421 * Change the behavior of the FairMQ channels. This is referred to as `ChannelConfigurationPolicy`.
422 * Change the command line options of the workflow driver.
423 
424 In all cases this is done by providing a:
425 
426 
427 ~~~~~~~{.cpp}
428 
429 void customize(...)
430 ~~~~~~~
431 
432 free function which takes as argument the group of policies to be applied to customise the behavior.

436 ## Support for analysis
437 
438 While not part of the initial design goal, we plan to extend DPL in order to support analysis. In particular we are evaluating a mode in which users can natively get a ROOT `RDataFrame` with an API similar to the `InputRecord` API.

440 ## Provenance dependent matching of inputs and outputs
441 
442 By default the Input and Outputs are matched solely by the signature of the data they contain. However sometimes it's desirable that this matching happens based on the history that a given message had, e.g. if it went through one path or another of a dataflow bifurcation. While this is not at the moment supported and you would have to use a separate data type for the two different origins, the usecase is acknowledged and will be addressed in a future revision of this document.
443 
444 ## Support for non compiled, non C++, based configuration
445 
446 At the moment the configuration of a topology is done by compiling a declarative description of the computation done in C++, and by running the resulting executable. This is however an implementation detail and the datamodel for the configuration description is meant to be agnostic from the language. We foresee both using interpreted configurations (e.g. via ROOT / cling) or configurations done through another language (e.g. go, javascript).

471 ## Data Sampling
472 
473 Data Sampling provides possibility to sample data in DPL workflows, basing on certain conditions ( 5% randomly, when payload is greater than 4234 bytes, etc.). The job of passing the right data is done by a data processor called `Dispatcher`. A desired data stream is specified in form of Data Sampling Policies, configured by JSON structures (example below).
474 ~~~~~~~
475 {
476  "id": "policy_example1", # name of the policy
477  "active": "false", # activation flag
478  "machines": [ # list of machines where the policy should be run (now ignored)
479  "aido2flp1",
480  "aido2flp2"
481  ],
482  "dataHeaders": [ # list of data that should be sampled
483  {
484  "binding": "clusters", # binding of the data in InputRecord
485  "dataOrigin": "TPC", # data origin in DataHeader
486  "dataDescription": "CLUSTERS" # data description in DataHeader
487  },
488  {
489  "binding": "tracks",
490  "dataOrigin": "TPC",
491  "dataDescription": "TRACKS"
492  }
493  ],
494  "subSpec": "0", # subspecification in DataHeader, use -1 for all
495  "samplingConditions": [ # list of sampling conditions
496  {
497  "condition": "random", # condition type
498  "fraction": "0.1", # condition-dependent parameter: fraction of data to sample
499  "seed": "2112" # condition-dependent parameter: seed of PRNG
500  }
501  ],
502  "blocking": "false" # should the dispatcher block the main data flow? (now ignored)
503 }
504 ~~~~~~~

542 ## Data Sampling Conditions
543 
544 The following sampling conditions are available. When more than one is used, a positive decision is taken when all the conditions are fulfilled.
545 - **DataSamplingConditionRandom** - pseudo-randomly accepts specified fraction of incoming messages.
546 
547 ~~~~~~~{.json}
548 
549 {
550  "condition": "random",
551  "fraction": "0.1",
552  "seed": "22222"
553 }
554 ~~~~~~~
555 - **DataSamplingConditionNConsecutive** - approves n consecutive samples in defined cycle. It assumes that timesliceID always increments by one.
556 
557 ~~~~~~~{.json}
558 
559 {
560  "condition": "nConsecutive",
561  "samplesNumber": "3",
562  "cycleSize": "100"
563 }
564 ~~~~~~~
565 - **DataSamplingConditionPayloadSize** - approves messages having payload size within specified boundaries.
566 
567 ~~~~~~~{.json}
568 
569 {
570  "condition": "payloadSize",
571  "lowerLimit": "300",
572  "upperLimit": "500"
573 }
574 ~~~~~~~
