3 # Status/Description of implementation
4 
5 At present, the `digitizer-workflow` executable is a demonstrator of
6 how we intend to do initiate and handle the processing of hits, coming from detector simulation.
7 
8 The digitizer-workflow currently demonstrates the transformation of hits into TPC digits using
9 realistic bunch crossing and collision sampling. We are also able to overlay hits from background and signal hit inputs.
10 
11 The main components of the digitizer-workflow are
12 
13 * The SimReader device:
14  - reading/analysing the given hit files
15  - performing the bunch crossing sampling and collision composition (stored in a collision context)
16  - initiating digitization/processing by communicating the collision context to processing devices
17 
18 * The TPC digitizier device:
19  - producing digits in continuous time for a given sector
20  - at present writes (or forwards) these digits in units of TPC drift times
21 
22 The digitizer-workflow executable is already somewhat configurable, both in terms of
23 workflow/topology options as well as individual device options. Some help is available
24 via
25 ```
26 digitizer-workflow --help
27 ```
28 Other features are demonstrated in he following section.
29 
30 # Feature example/Usage
31 
32 Let's assume we have a background hit file `o2sim_bg.root` generated
33 by the O2 simulation with
34 ```
35 o2sim -n 20 -g SOMEBACKGROUNDEVENTGENERATOR -m [detectors] -o o2sim_bg.root
36 ```
37 
38 Similar for a signal file `o2sim_sg.root`
39 ```
40 o2sim -n 50 -g SOMESIGNALEVENTGENERATOR -m [detectors] -o o2sim_sg.root
41 ```
42 
43 1. **How can I digitize all sectors for the given background event?**
44  ```
45  digitizer-workflow -b --simFile o2sim_bg.root
46  ```
47  This will run as many TPC digitizer processors as there are logical CPU cores on your machine in parallel.
48  (Note that depending on your available memory, this might cause problems as the digitization needs lots of memory; It might be safer to start with a small number of workers as indicated under point 3.).
49 
50 2. **How can I only digitize sectors TPC sectors 1 + 2 for the given background event?**
51  ```
52  digitizer-workflow -b --tpc-sectors=1,2 --simFile o2sim_bg.root
53  ```
54 
55 3. **How can I digitize sectors 1-8 using only 2 TPC digitizer devices?**
56  ```
57  digitizer-workflow -b --tpc-lanes=2 --tpc-sectors=1,2,3,4,5,6,7,8 --simFile o2sim_bg.root
58  ```
59 
60 4. **How can I digitize a total of 100 sampled collisions merging background and signal hits for TPC sector 1?**
61  ```
62  digitizer-workflow -b --tpc-sectors=1 --simFile o2sim_bg.root --simFileS o2sim_sg.root -n 100
63  ```

65 # Missing things/Improvements to come
66 
67 At present the digitizer write individual digit files for each sector with names `tpc_digi_22_...`.
68 It is planned asap to make this more configurable and to outsource the writing to ROOT files in a different device.
69 
70 Configuration of the workflow via environment variables is going to be substituted via a proper mechanism once this
71 is implemented by DPL.
72 
73 Digitizers for other detectors shall be added.
74 
75 The polay distribution should be commicated via CDB or some init mechanism.
