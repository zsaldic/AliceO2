1 # Configurable Parameters
2 
3 This is a short README for configurable parameters as offered by
4 the ConfigurableParameter class.
5 
6 # Introduction
7 
8 The ConfigurableParameter class implements the demand
9 * to have simple variables (under a compound namespace) be declared as 'knobs'
10 of an algorithm in order to be able to change/configure their value without recompilation.
11 * to have such declared variables be automatically registered in a parameter database or manager instance.
12 * to be able to configure/change the parameter values in a textual way (for instance from the command line)
13 * to provide automatic serialization / deserialization techniques (to text files or binary blobs) -- for instance to load from CCDB or to pass along parameter snapshots to other processing stages.
14 * to keep track of who changes parameters (provenance tracking).
15 
16 # Example / HowTo:
17 
18 Imagine some algorithms `algorithmA` depends on 2 parameters `p1` and `p2` which you want to be able to configure.
19 
20 You would do the following steps:
21  1. Declare a parameter class listing the parameters and their default values.
22  ```c++
23  struct ParamA : ConfigurableParamHelper<ParamA> {
24  int p1 = 10;
25  double p2 = 1.23;
26  // boilerplate stuff + make parameters known under key "A"
27  O2ParamDef(ParamA, "A");
28  };
29  ```
30  2. Put
31  ```
32  O2ParamImpl(ParamA);
33  ```
34  in some source file, to generate necessay symbols needed for linking.
35  3. Access and use the parameters in the code.
36  ```c++
37  void algorithmA() {
38  // get the parameter singleton object
39  auto& pa = ParamA::Instance();
40  // access the variables in your code
41  doSomething(pa.p1, pa.p2);
42  }
43  ```
44 
45 Thereafter, the parameter `ParamA` is automatically registered in a parameter registry and can be read/influenced/serialized through this. The main influencing functions are implemented as static functions on the `ConfigurableParam` class. For example, the following things will be possible:
46 * get a value by string key, addressing a specific parameter:
47  ```
48  auto p = ConfigurableParam::getValue<double>("A.p2");
49  ```
50  where the string keys are composed of a primary key and the variable name of the parameter.
51 * set/modify a value by a key string
52  ```
53  ConfigurableParam::setValue<double>("A.p2", 10);
54  ```
55  where the string keys are composed of a primary key and the variable name of the parameter.
56  Side note: This API allows to influence values from a string, for instance obtained from command line:
57  ```
58  ConfigurableParam::fromString("A.p2=10,OtherParam.a=-1.");
59  ```
60  The system will complain if a non-existing string key is used.
61 
62 * serialize the configuration to a ROOT snapshot or to formats such as JSON or INI
63  ```
64  ConfigurableParam::toINI(filename);
65  ConfigurableParam::toJSON(filename); // JSON repr. of param registry
66  ConfigurableParam::toCCDB(filename); // CCDB snapshot of param registry
67  ```
68 * retrieve parameters from CCDB snapshot
69  ```
70  ConfigurableParam::fromCCDB(filename);
71  ```
72 * **Provenance tracking**: The system can keep track of the origin of values. Typically, few stages of modification can be done:
73  - default initialization of parameters from code (CODE)
74  - initialization/overwritting from a (CCDB) snapshot file
75  - overriding by user during runtime (RT)
76 
77  The registry can keep track of who supplied the current (decisive) value for each parameter: CODE, CCDB or RT. If a parameter is not changed it keeps the state of the previous stage.
78 
79 # Further technical details
80 
81 * Parameter classes are **read-only** singleton classes/structs. As long as the user follows the pattern to inherit from `ConfigurableParamHelper<T>` and to use the macro `O2ParamDef()` everything is implemented automatically.
82 * We use the `ROOT C++` introspection facility to map the class layout to a textual configuration. Hence ROOT dictionaries are needed for parameter classes.
83 * BOOST property trees are used for the mapping to JSON/INI files but this is an internal detail and might change in future.
84 
85 # Limitations
86 
87 * Parameter classes may only contain simple members! Currently the following types are supported
88  * simple pods (for example `double x; char y;`)
89  * std::string
90  * fixed size arrays of pods using the ROOT way to serialize:
91  ```c++
92  static constexpr int N=3;
93  double array[N] = {1, 2, 3}; //[N] -- note the [N] after //!!
94  ```
95  * array parameters need to be addressed textual with an index operator:
96  ```
97  ConfigurableParam::fromString("ParamA.array[2]=10");
98  ```
99  It is planned to offer more flexible ways to set arrays (see below).
100  * there is currently no support for pointer types or objects. Parameter classes may not be nested.
101 * At the moment serialization works on the whole paramter registry (on the whole set of parameters). Everything is written to the same file or snapshot.
102 
103 # Wish list / planned improvements
104 
105 * Offer a more flexible way to set array types (to represent them in strings), for example
106  ```c++
107  ConfigurableParam::fromString("ParamA.array = {5, 6, 7}");
108  ```
109 * Offer a better more flexible way to read from CCDB.
110  * read from complete snapshots (give single file name) **or** read from different files reciding in paths corresponding to key/namespace.
111 * Of a more flexible way to serialize, for example allowing to output configuration to different files.
112 * Be able to change the configuration from a text file (next to current possibility from command line)
113 * support for few important stl containers (std::array, std::vector)
114 * a better way to define stages (CODE, CCDB, RT) and to transition between them; potentially more allowed stages
115 * take away more boilerplate: Automatic creation of dictionaries for parameter classes.
