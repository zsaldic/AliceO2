3 ## MCStepLogger
4 
5 Detailed debug information about stepping can be directed to standard output using the `LD_PRELOAD` env variable, which "injects" a
6  special logging library (which intercepts some calls) in the executable that follows in the command line.
7 
8 
9 ~~~~~~~{.sh}
10 
11 LD_PRELOAD=path_to/libMCStepLogger.so o2sim -m MCH -n 10
12 ~~~~~~~
13 
14 
15 ~~~~~~~
16 [MCLOGGER:] START FLUSHING ----
17 [STEPLOGGER]: did 28 steps
18 [STEPLOGGER]: transported 1 different tracks
19 [STEPLOGGER]: transported 1 different types
20 [STEPLOGGER]: VolName cave COUNT 23 SECONDARIES 0
21 [STEPLOGGER]: VolName normalPCB1 COUNT 3 SECONDARIES 0
22 [STEPLOGGER]: ----- END OF EVENT ------
23 [FIELDLOGGER]: did 21 steps
24 [FIELDLOGGER]: VolName cave COUNT 20
25 [FIELDLOGGER]: ----- END OF EVENT ------
26 [MCLOGGER:] END FLUSHING ----
27 ~~~~~~~
28 
29 The stepping logger information can also be directed to an output tree for more detailed investigations.
30 Default name is `MCStepLoggerOutput.root` (and can be changed
31 by setting the `MCSTEPLOG_OUTFILE` env variable).
32 
33 
34 ~~~~~~~{.sh}
35 
36 MCSTEPLOG_TTREE=1 LD_PRELOAD=path_to/libMCStepLogger.so o2sim ..
37 ~~~~~~~
38 
39 Finally the logger can use a map file to give names to some logical grouping of volumes. For instance to map all sensitive volumes from a given detector `DET` to a common label `DET`. That label can then be used to query information about the detector steps "as a whole" when using the `StepLoggerTree` output tree.
40 
41 
42 ~~~~~~~{.sh}
43 
44 > cat volmapfile.dat

46 normalPCB2 MCH
47 normalPCB3 MCH
48 normalPCB4 MCH
49 normalPCB5 MCH
50 normalPCB6 MCH
51 centralPCB MCH
52 downroundedPCB MCH
53 uproundedPCB MCH
54 cave TheCavern
55 
56 > MCSTEPLOG_VOLMAPFILE=path_to_/volmapfile.dat MCSTEPLOG_TTREE=1 LD_PRELOAD=path_to/libMCStepLogger.so o2sim ..
57 
58 > root -b MCStepLoggerOutput.root
59 root[0] StepLoggerTree->Draw("Lookups.volidtomodule.data()");
60 ~~~~~~~

86 ## MCStepLogAnalysis
87 
88 Information collected and stored in `MCStepLoggerOutput.root` can be further investigated using the excutable `mcStepAnalysis`. This executable is independent of the simulation itself and produces therefore no overhead when running a simulation. 2 commands are so far available (`analyze`, `checkFile`) including useful help message when typing
89 
90 ~~~~~~~{.sh}
91 
92 mcStepAnalysis <command> --help
93 ~~~~~~~
94 
95 ### File formats
96 
97 2 file formats having a standardised structure play a role. On one hand, these are the files produced by the step logging which are the input files for the analysis as explained in the following. On the other hand, each analysis produces an output file containing histograms along with some meta information. Sanity and the type of the file can be checked via
98 
99 ~~~~~~~{.sh}
100 
101 mcStepAnalysis checkFile -f <FileToBeChecked>
102 ~~~~~~~
103 ### Analysing the steps
104 
105 The basic command containing all required parameters is
106 
107 ~~~~~~~{.sh}
108 
109 mcStepAnalysis analyze -f <MCStepLoggerOutputFile> -o <parent/output/dir> -l <label>
110 ~~~~~~~
111 where
112 * `-f <MCStepLoggerOutputFile>` passes the input file produced with the `MCStepLogger` as explained above (default name is `MCStepLoggerOutput.root`)
113 * `-o <parent/output/dir>` provides the top directory for the analysis output (if this does not exist, it is created automatically)
114 * `-l <label>` adds a label, e.g. for plots produced later.
115 
116 A `ROOT` file at `parent/output/dir/MetaAnalysis/Analysis.root` is produced containing all histograms as well as important meta information. Histogram objects are derived from `ROOT`s `TH1` classes.
117 
118 ### Further processing of analysis files
119 
120 Files produced as described before can be investigated further or used to plot the histograms therein. The interface to read these files is the class `AnalysisFile` and histograms can be requested by their names.
121 
122 ~~~~~~~{.c}
123 ++
124 // somthing...
125 #include "MCStepLogger/AnalysisFile.h"
126 // some code
127 AnalysisFile af;
128 af.read("path/to/file.root");
129 // print meta info
130 af.printMetaInfo();
131 // get a histogram by name (program will exit if name not found)
132 TH1& histogram = af.getHistogram("nameOfHistogram");
133 // if you know the underlying object is, e.g. of derived class TH1D and you really need that you can do
134 TH1D& otherHistogramCasted = af.getHistogram<TH1D>("nameOfOtherHistogram");
135 // where the program will safely exit immediately in case the casting was not successful
136 //...
137 // modify histogram or do other things
138 //...
139 //to save changes made to histograms do
140 af.write("path/to/output/file.root");
141 ~~~~~~~
142 ### Additional information about the `MCAnalysisManager`
143 
144 There is the staic method `MCAnalysisManager::Instance()` which returns a reference to a static instance. So always make sure you don't copy it but get the reference in case you want to work with that instance on a global scope, i.e.
145 
146 ~~~~~~~{.c}
147 ++
148 // some code
149 auto& anamgr = MCAnalysisManager::Instance();
150 ~~~~~~~
151 ### Additional information about the analysis objects
152 
153 Histograms which should be written to disk in an analysis are managed by `MCAnalysisFileWrapper` objects. These also make sure that no histogram is created twice. Therefore, all of these histograms should be created like `T* myHisto = MCAnalysis::getHistogram<T>(...)` where the template parameter `T` must be a class deriving from ROOT's `TH1`. It then returns a pointer to the desired object. Managing histograms not on the level of an analysis also enables for requesting histograms from another analysis. In that way one can write a custom analysis for a specific use case but can still ask for e.g. for a histogram from the `BasicMCAnalysis` to derive some additional and more generic information about a simulation run. Hence, never manually delete an object obtained like this.
154 
155 ### Comparing analysis values to reference reference values
156 
157 For the `BasicMCAnalysis` there is a small test suite to compare the obtained values from a simulation run to reference values contained in a JSON file. So far, that is a prototype only caring about the total number of steps and total number of tracks obtained in the simulation. The `JSON` file looks as follows
158 
159 ~~~~~~~{.json}
160 
161 {
162  "analysisName": "BasicMCAnalysis",
163  "nSteps": [absoluteNumberOfSteps, relativeTolerance],
164  "nTracks": [absoluteNumberOfTracks, relativeTolerance]
165 }
166 ~~~~~~~
167 The test is steered via
168 
169 ~~~~~~~{.sh}
170 
171 runTestBasicMCAnalysis -- <path/to/Analysis.root> <reference.json>
172 ~~~~~~~
173 Note, that the test does not know anything about the settings of the simulation run, i.e. there are no information about the primary generator of the transport engine etc. The user has to make sure to apply this coherently.

175 ### Writing/running a custom analysis
176 
177 Although providing already a number of different observables, users might want to add custom observables for their analysis. To do so, a directory for custom analyses has to be created where analysis macros can be provided and loaded at run-time. Note, that only the basic analysis is actually contained in the compiled code. One of the main reasons for that is to enable for a coherent comparison between different points in the git history. However, if you feel like there is an important observable missing, feel free to report that.
178 
179 The logic of adding a custom analysis is very similar to that of `Rivet` and the general workflow should look familiar in any case. Say, your analysis macro directory is `$ANALYSIS_MACROS/` where you have your macro `mySimulationAnalysisc.C` (don't place any other files there since these cannot be read...). A skeleton looks as follows, also containing more information on how and why things are implemented like they are:
180 
181 
182 ~~~~~~~{.c}
183 ++
184 // myMCStepAnalysis.C
185 //
186 // headers you need
187 
188 /* Your analysis class has to derive from the base o2::mcstepanalysis::MCAnalysis. All
189  * methods to be implemented are mentioned below.
190  */
191 class MyMCStepAnalysis : public MCAnalysis
192 {
193  public:
194  /* You don't need a fancy constructor, that's it. The base constructor takes care
195  * of registering this analysis to the global o2::MCStepAnalysis::MCAnalysisManager
196  * object. An MCAnalysis object cannot exist without the MCAnalysisManager.
197  */
198  MySimulationAnalysis()
199  : MCAnalysis("MyMCStepAnalysis")
200  {}
201 
202  /* There are 3 methods covering the analysis: initialize, analyze and finalize.
203  * The first two need to be overriden in any case.
204  * These methods are called by the MCAnalysisManager. Other custom methods
205  * can hence only be used for class-internal purposes only.
206  */
207 
208  /* All histograms used in the analysis must be defined here. This is done using
209  * the method MCAnalysis::getHistogram<T>(...) where the template paramter T has to
210  * be a deriving class of ROOT's TH1. The histogram pointers are the managed
211  * globally for further processing, saving, plotting etc. It is also taken care
212  * of the deletion of the pointers. Do not attempt to delete histograms obtained by
213  * MCAnalysis::getHistogram<T>(...). Histograms are uniquely identified by there
214  * name and an MCAnalysis will stop immediately at the initialisation stage if
215  * two histograms with the same name are detected.
216  */
217  void initialize() override {
218 
219  // monitor calls to the magnetic field per volume
220  histMyFirstObservable = getHistogram<TH1D>("myFirstObservable", 1, 0., 1.);
221  // monitor the steps in the r-z plane
222  histMySecondObservable = getHistogram<TH2D>("mySecondObservable", 1, 0., 1., 2, 0., 2.);
223  // more histograms?! Other things to do?
224 
225  }
226 
227  /* This method is used to extract the step information in order to fill histograms
228  * accordingly.
229  */
230  void analyze(const std::vector<StepInfo>* const steps,
231  const std::vector<MagCallInfo>* const magCalls) override {
232 
233  // loop over mag field calls and match to step ID
234  for(const auto& call : *magCalls) {
235  auto step = steps->operator[](call.stepid);
236  mAnalysisManager->getLookupVolName(step.volId, volName);
237  histMyFirstObservable->Fill(volName.c_str(), 1.);
238  }
239 
240  // loop over steps and get z-position and other things as you like
241  for(const auto& step : *steps) {
242  histMySecondObservable->Fill(step.z, std::sqrt( step.x*step.x + step.y*step.y ));
243  // ...and do more, as you like
244  }
245  }
246 
247  /* the finalyze method can be used to do necessary adjustments like scaling or adding
248  * histograms or whatever must be done to them.
249  */
250  void finalize() override {
251  histMyFirstObservable->Scale( 1. / histMyFirstObservable->GetEntries() );
252  // ...and more...
253  }
254 
255  private:
256  // you might want to provide some useful methods for internal use...
257 
258  private:
259  TH1D* histMyFirstObservable;
260  TH2D* histMySecondObservable;
261  // other histogram pointers and/or members for internal usage
262  };
263 
264 /* The hook to bring your analysis into existence so that the MCAnalysisManager
265  * knows. Don't be scared about the way the pointer of the analysis is created. It is all
266  * taken care of by the MCAnalysisManager since the base MCAnalysis constructor
267  * automatically registeres itself to the MCAnalysisManager. So don't worry about
268  * pointers flying around.
269  */
270 void declareAnalysis() {
271  new MySimulationAnalysis("mySimulationAnalysis");
272 }

274 After having this, you are ready to include this in the analysis run by typing
275 
276 ~~~~~~~{.sh}
277 
278 runMCAnalysis analyze -f <MCStepLoggerOutputFile> -o <parent/output/dir> -l <label> -d $ANALYSIS_MACROS -a mySimulationAnalysis
279 ~~~~~~~
280 where now
281 * `-d $ANALYSIS_MACROS` points the executable to the directory of where your macros are located
282 * `-a mySimulationAnalysis` tells which analysis to load. In case you have more analyses in that directory you want to load, just append the names of all analyses you want to run.
283 The output of the custom analysis is written to `parent/output/dir/mySimulationAnalysis/` and that's it.
